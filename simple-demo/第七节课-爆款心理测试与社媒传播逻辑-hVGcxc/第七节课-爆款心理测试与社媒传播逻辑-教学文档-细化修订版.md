# 第七节课：爆款心理测试与社媒传播逻辑 教学文档（细化修订版）

## 本节内容总结

这节课的重点不是演示某个后台怎么点，而是把“一个测试类产品为什么能传播”拆成可以直接复用的方法。整节课可以按下面几条主线理解：

1. 先用一个真实项目说明，为什么要把 `PRD`、规则、任务、进度拆成很多小文档，方便 `AI` 协作开发。
2. 再解释为什么心理测试天生适合传播，核心不在题目数量，而在用户心理和可展示结果。
3. 接着把题目设计、结果页设计、传播关键时刻、平台差异拆开，形成一套可复用的内容打法。
4. 最后再用“热门 vs 不热门”的对照方式，帮助自己复盘一个测试内容到底差在哪里。

如果后面要做的是：

- 心理测试
- 人格测评
- 社交互动类小程序
- 话题型内容产品

这一节课最值得反复看的，不是某一页结论，而是“从产品结构到传播结构”的整套思路。

## 详细操作步骤

### 小tips：先把项目文档拆成 AI 容易读取的小文件
【视频回放时点00：03：00】

课程开头先展示了一个真实项目，用它来说明为什么现在做产品时，文档不能只写一份总文件，而要拆成多个小文件。

先理解这个动作的目标：

1. 不是为了把文档写得更好看
2. 而是为了让 `AI` 每次接任务时，能更快读到当前真正相关的上下文
3. 同时降低跑偏、漏规则、忘约束的概率

第一步，先把项目根目录和文档目录分开看。  
从演示可以看到，项目不是只放代码，而是同时保留：

1. `backend`
2. `frontend`
3. `doc`
4. 若干规则文件和说明文件

![小tips截图A](/Users/shihui/Desktop/玩玩编程/生产教学文档/simple-demo/hVGcxc/work/refined_samples/frame_240.jpg)

第二步，把总规则单独写出来。  
课程里展示了 `CLAUDE.md` 这类总规则文件，里面会统一说明：

1. 项目概述
2. 技术栈
3. 项目结构
4. 执行约束
5. 编码规范

这类文件的作用不是替代所有文档，而是给 `AI` 一个总入口，保证它每次开始工作前先读到最关键的全局规则。

![小tips截图B](/Users/shihui/Desktop/玩玩编程/生产教学文档/simple-demo/hVGcxc/work/refined_samples/frame_180.jpg)

第三步，把“待确认问题”和“产品规则”拆成独立文件。  
演示里专门把 `PRD待确认问题清单` 这类文件单独放出来，这样做有两个直接好处：

1. 模糊问题不会散落在聊天记录里
2. 每次有新问题，都能在固定文档里继续补充和确认

如果后面需要和 `AI` 来回讨论产品方向，这个文件会比单纯在聊天窗口里反复描述更稳。

![小tips截图C](/Users/shihui/Desktop/玩玩编程/生产教学文档/simple-demo/hVGcxc/work/refined_samples/frame_360.jpg)

第四步，把任务计划单独做成 `task_plan.md`。  
这里不是写一句“做首页、做后台、做支付”就结束，而是要把任务继续拆细，拆到：

1. 阶段目标
2. 当前完成状态
3. 哪些已经完成
4. 哪些还没完成
5. 下一步应该衔接什么

课程里强调的重点就是：任务拆得越细，`AI` 越不容易偏离开发要求。

![小tips截图D](/Users/shihui/Desktop/玩玩编程/生产教学文档/simple-demo/hVGcxc/work/refined_samples/frame_480.jpg)

第五步，把前端文档继续按页面和模块拆开。  
演示里能看到前端目录下面不是一个大文档，而是拆成：

1. 设计系统
2. 导航
3. 页面体系
4. 各个业务页面的独立文档

这样做的原因非常直接：

1. `AI` 不需要每次把整套说明都读完
2. 读到当前页面对应的文档，就能开始工作
3. 前端、后端、文档协作者之间的边界也更清晰

![小tips截图E](/Users/shihui/Desktop/玩玩编程/生产教学文档/simple-demo/hVGcxc/work/refined_samples/frame_660.jpg)

注意：

- 不要只保留一个超长 `md` 文档，否则上下文很快就会被无关内容占满。
- 每次做完一轮任务，记得同步更新 `notes`、`progress`、`task_plan`。
- 如果规则会长期影响开发，比如字段命名、接口风格、禁止事项，最好单独成文，不要只写在聊天记录里。

### 步骤2：先理解为什么心理测试天然适合传播
【视频回放时点00：15：00】

课程里先给出了四个很关键的触发点，这四点决定了为什么心理测试比很多普通工具更容易扩散：

1. `自我认同`
用户天然会被“我是什么样的人”吸引。
2. `关系确认`
当题目涉及“我和谁更像”“我和 TA 是否合得来”时，就会自然触发分享和互动。
3. `情绪出口`
焦虑、好奇、娱乐感都能被测试形式很好地承接。
4. `谈资价值`
结果一旦能晒、能聊、能引发回应，就具备二次传播能力。

这里最重要的一句判断是：
用户做测试，不只是为了答完题，而是为了拿到一个可展示、可表达、可分享的结果。

注意：

- 如果结果只是一句平淡描述，传播力会很弱。
- 如果结果能让用户“觉得在说我”，传播欲会明显变强。

### 步骤3：题目设计时，先守住“短、准、爽”三条底层原则
【视频回放时点00：25：00】

开始设计题目时，先不要急着堆很多题，而是先检查底层原则是否成立：

1. `短`
题量不要过重，单题理解成本要低，用户能迅速进入状态。
2. `准`
题目和选项要贴近真实场景，不要用太学术、太抽象的表达。
3. `爽`
用户做题时要不断产生“这题说的就是我”的感觉。

题目示例和差题示例的对比，说明了一个很关键的点：

1. 好题不是专业，而是代入快
2. 差题往往不是错误，而是过于抽象、过于绕

如果题目读起来像问卷调查，用户很容易中途退出；  
如果题目像是在讲用户自己的日常场景，用户会更容易一直往下做。

### 步骤4：结果页一定要做成“用户愿意晒”的页面
【视频回放时点00：30：00】

很多测试内容前面做得不错，但最后没有传播起来，问题往往不在题目，而在结果页太平。

一页可传播的结果页，至少要有下面几块：

1. `标签名`
让用户一眼知道自己属于哪个类型。
2. `一句解释`
用一句人能立刻懂的话解释用户当前状态。
3. `社交文案`
给用户现成可复制、可发圈、可转发的表达。
4. `结构化结果`
例如情绪描述、优势提醒、关系建议、可截图区域。

这一步的核心标准不是“看起来完整”，而是：
用户看到以后，会不会产生“这个我想发给别人看”的冲动。

注意：

- 结果不要只给分数。
- 结果不要写得像诊断报告。
- 结果越方便截图，越容易带来传播。

![步骤4截图](/Users/shihui/Desktop/玩玩编程/生产教学文档/simple-demo/hVGcxc/work/samples/frame_1800.jpg)

### 步骤5：复盘案例时，按完整链路去拆，不要只盯某一个点
【视频回放时点00：40：00】

课程里用一张表把案例拆成了完整链路。以后你自己复盘一个测试内容时，也可以直接照着这五项拆：

1. `选题`
是不是足够贴热点、贴人格、贴关系、贴生活情境。
2. `标题`
是不是一眼就能代入，能激发点击。
3. `结果`
是不是具备截图价值和讨论价值。
4. `传播`
是不是设计了评论区互动、朋友互测、二次扩散。
5. `转化`
是不是还能自然承接到产品动作或商业化入口。

这样拆的好处是：

1. 不会把“火”简单归因给一个标题
2. 能看到内容从点击到转化的完整闭环
3. 更容易找出自己内容真正薄弱的地方

### 步骤6：做传播设计时，重点盯住四个关键时刻
【视频回放时点00：50：00】

如果想让测试类产品真正扩散，不要只想着“发布出去”，而要盯住用户经历的四个关键时刻：

1. `看到标题的前 3 秒`
会不会点进去。
2. `做到前几题时`
愿不愿意继续往下做。
3. `看到结果页时`
会不会想截图。
4. `分享出去以后`
别人会不会接着互动、留言、参与。

课程里这一页的核心提醒是：
传播不是“发出去”就结束，传播是别人愿意接着传。

所以在设计内容时，要反过来问：

1. 标题能不能勾住人
2. 中间会不会掉人
3. 结果有没有表达欲
4. 分享出去以后有没有互动空间

![步骤6截图](/Users/shihui/Desktop/玩玩编程/生产教学文档/simple-demo/hVGcxc/work/samples/frame_3000.jpg)

### 步骤7：换平台时，不要直接复制同一套表达
【视频回放时点01：00：00】

同一个测试主题，如果放到不同平台上，必须重新包装。课程里这一页把差异说得很清楚：

1. `抖音`
节奏快、情绪驱动强，前几秒必须有强钩子。
2. `小红书`
更看重生活方式、人设和真实场景，不能空泛。
3. `视频号`
更适合把观点讲完整，重信任和逻辑。
4. `公众号`
适合深度拆解和案例复盘，内容结构要清楚。

这一步最重要的不是记平台名，而是记一个原则：
不要把同一份内容原样复制到所有平台。

注意：

- 平台不同，用户耐心不同
- 平台不同，标题风格不同
- 平台不同，适合的内容颗粒度也不同

### 步骤8：最后用“热门 vs 不热门”的对照方式做自检
【视频回放时点01：10：00】

课程最后这张对照表特别适合直接拿来做复盘模板。以后一个测试内容发出去没有起量，不要先说“平台不给流量”，可以先按这几项反查：

1. `标题`
有没有足够强的代入感和冲突感。
2. `题目`
是不是足够具体、足够好懂。
3. `结果`
有没有截图价值、分享价值、讨论价值。
4. `传播`
有没有互动设计，而不是发完就结束。
5. `转化`
有没有后续承接，而不是只有一波表面流量。

如果这几项里面连续几项都偏弱，那么内容很可能不是“没人看到”，而是“看到了也没有继续传播的动力”。

## 课后可直接复用的检查清单

如果准备自己做一个测试类产品，可以直接按下面顺序自查：

1. 是否已经把总规则、任务计划、待确认问题、前后端文档拆成独立文件
2. 选题是否同时具备自我认同、关系互动、情绪出口或谈资价值
3. 题目是否符合“短、准、爽”
4. 结果页是否具备标签、一句话解释、社交文案和截图价值
5. 传播链路是否覆盖点击、继续答题、截图、互动四个关键时刻
6. 不同平台是否已经分别做了对应包装
7. 发布后能否用“热门 vs 不热门”对照表继续复盘

把这几项做扎实，这节课讲的方法就真正能落到自己的产品里。
