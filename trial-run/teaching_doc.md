## 说明

这是一版基于课程视频抽样帧生成的试跑稿，当前优先验证这套 skill 能否把视频内容整理成“学生可读”的教学文档。

这版文档已经做了重构：

- 保留了课程主线
- 合并了重复演示
- 把“老师边讲边绕”的部分改成了更适合学员执行的顺序

## 本节目标

本节课的核心不是一次性把小程序和 Apple 后台所有细节都做到完美，而是先把完整发布路径理清楚，并跑通一版可提交、可验证的基础流程。

本节围绕四件事展开：

1. 明确个人主体上架时的限制与策略
2. 在微信平台侧处理小程序名称与基础资料
3. 在微信开发者工具中把项目跑起来并完成预览验证
4. 在 App Store Connect 中理解版本素材与后台配置页

## 前置准备

开始前，建议先准备好下面这些材料：

- 一个可以运行的小程序项目
- 备用的小程序名称方案，避免重名
- 项目简介、功能描述、截图素材
- 已可登录的微信公众平台账号
- 已可登录的 App Store Connect 账号
- 微信开发者工具

## 第一部分：先判断项目是否适合个人主体提交

老师在前半段先强调了一个很重要的判断原则：如果你现在是个人主体，不要一开始就做能力太重、审核风险太高的版本。

更稳的做法是：

- 优先做内容展示、工具类、简单交互类场景
- 暂时避开复杂支付或高风险开放能力
- 名称、简介、截图、功能说明保持一致
- 先做一个简单可审核版本，后续再迭代升级

对学生来说，这一段最重要的结论不是某一个按钮怎么点，而是审核策略：

**先上线，再升级。**

如果一开始就把功能做得太满，往往会在审核、命名或资料一致性上反复卡住。

## 第二部分：在微信公众平台处理名称与基础资料

从课程画面看，老师演示了微信平台里的名称修改流程，并且触发了“名称重复”的典型问题。

这里可以整理成一套更清晰的执行步骤：

### 1. 进入小程序基础资料或主体资料相关页面

先找到小程序名称可编辑的入口，准备修改名称。

### 2. 输入新的小程序名称

命名时不要只考虑“自己喜欢什么名字”，而要考虑：

- 是否容易与已有名称重复
- 是否清楚表达产品用途
- 是否容易和简介、截图、功能说明对应上

### 3. 处理平台的重名或合规提示

如果平台提示名称重复，不要硬顶着继续提交。此时应该：

- 准备第二、第三套备用名称
- 尽量让名称更具体
- 避免模糊、误导或过度泛化的词

### 4. 再继续进入下一步

这一段在正式文档里建议保留一张“名称重复提示”的截图，因为它很能帮助学生建立预期：命名问题不是个例，而是提交前的常见卡点。

## 第三部分：下载安装微信开发者工具

老师中段展示了微信开发者文档的工具下载页面，这一步非常适合作为操作文档里的一个独立步骤。

建议写成：

### 1. 打开微信开发者文档

进入开发者文档后，定位到“工具 / 下载”。

### 2. 下载与你系统匹配的稳定版

例如课程里展示的是 `下载 macOS ARM64` 入口。你在正式操作时，应根据自己的系统版本选择对应安装包。

### 3. 安装完成后再进入项目调试

不要在工具尚未安装完成时继续后面的项目导入与预览步骤，否则学生会在环境问题上卡住。

这一部分最适合配一张“下载页入口”的定位截图。

## 第四部分：在微信开发者工具中跑通项目

从后续画面可以确认，老师已经进入微信开发者工具，并在左侧资源管理器、中间代码区、右侧模拟器之间来回操作。

对学生来说，真正该记住的是下面这条顺序：

### 1. 导入项目

先把项目目录导入到微信开发者工具中，确保工具能识别项目结构。

### 2. 先看能不能跑起来

此时不要急着做精细优化，先确认右侧模拟器是否能正常显示页面。

### 3. 再做页面或交互层面的修正

如果页面已经能打开，就继续：

- 调整代码
- 观察模拟器反馈
- 修复明显报错或布局异常

### 4. 做预览与基础验证

至少要保证：

- 页面能打开
- 核心功能能跑通
- 展示内容和你要提交的产品描述一致

这一段课程里还出现了不同页面或 Demo 的演示画面，说明老师在强调的不是某一个写法，而是一个原则：

**先把流程跑通，再打磨细节。**

## 第五部分：理解 Apple 后台版本页要配置什么

课程后段进入了 App Store Connect，并停在 iOS App 版本页面的素材区域。

这里在教学文档里可以重构成：

### 1. 进入对应版本页

找到你当前准备提交的 iOS 版本。

### 2. 切换到对应设备素材区域

例如 `iPhone`、`iPad` 等素材位。

### 3. 准备并上传预览图或所需媒体素材

从课程画面看，页面上已经出现了 `选择文件` 入口。这说明这一段重点是让学生知道：

- 版本页不仅是看信息
- 这里还要补上传素材
- 素材尺寸、数量和内容都要符合平台要求

### 4. 保证后台素材与前面提交思路一致

如果你前面的小程序命名、功能描述、产品定位和这里准备的素材不一致，后续会造成提交逻辑割裂。

因此正式文档里最好把“资料一致性”再次强调一遍。

## 本节最重要的知识重构

老师的视频是边讲边操作的，但整理成学生版文档后，真正应该记住的是下面这条主线：

1. 先确认审核策略与主体限制
2. 再处理名称和基础资料
3. 然后安装并准备开发环境
4. 接着在开发者工具中跑通项目
5. 最后再回到 Apple 后台补齐版本发布所需内容

这个顺序比“跟着老师口头顺序逐句记”更适合学员执行。

## 常见错误

### 错误 1：一开始就追求完整功能

结果是审核风险高、资料难统一、版本迟迟交不上去。

### 错误 2：名称、简介、截图各说各话

这会直接影响提交质量，也容易让审核逻辑变得不清晰。

### 错误 3：开发工具和项目环境没准备好就往后走

学生会误以为自己不会操作，实际只是环境没先跑通。

### 错误 4：把 Apple 后台当成最后随便补一下的页面

实际上它和前面的命名、素材、产品说明是联动的。

## 课后作业

根据课程最后的总结页，课后更合理的交付应该是：

- 完成一个本地可运行的小程序 Demo
- 准备功能说明
- 准备命名方案
- 提供测试截图
