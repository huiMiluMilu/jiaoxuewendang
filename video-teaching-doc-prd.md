# 视频生成教学文档 Skill PRD

## 1. 文档信息

- 产品名称：视频生成教学文档 Skill
- 文档类型：PRD
- 当前版本：V0.1
- 更新时间：2026-03-31
- 负责人：Codex 协作草稿

## 2. 背景与问题

当前目标是把课程回放、软件录屏、老师演示视频，半自动整理成可交付的教学文档。

现有直觉流程容易从“抽帧”开始，但这会带来两个问题：

- 容易误判课程主线。抽样帧只能看到局部画面，未必能还原老师真正讲的重点。
- 容易把截图当成主事实来源，导致步骤判断、点击标注、知识重构都被画面采样误导。

从最近一次试跑看，课程实际重点是“开发微信小程序”，而基于抽帧的早期判断出现了偏差。这说明产品的第一事实来源应当是语言内容，其次才是画面证据。

因此，本产品的核心方向调整为：

先把老师的语言整理成可编辑文字，再基于文字和画面联合完成步骤拆分、截图选择、点击标注和教学文档初稿。

## 3. 产品目标

### 3.1 总目标

构建一个可复用的 Skill，把视频内容半自动转换为结构化教学文档，并支持后续增强为高质量、多受众、多风格输出。

### 3.2 阶段目标

#### 第一阶段：半自动

先做到以下能力，并以“可人工修改”为主要成功标准：

- 自动转写
- 自动分步骤
- 自动选截图
- 自动标点击点
- 自动生成初稿

#### 第二阶段：结构化增强

在第一阶段稳定后，增加：

- 主流程 / 说明 / FAQ 分离
- 混乱讲解重构
- 一步一动作
- 预期结果与注意事项补全

#### 第三阶段：高质量版本

在前两阶段稳定后，继续增强：

- 多风格输出
- 面向不同受众的版本
- 更稳定的 UI 元素识别
- 更强的点击对象命名

## 4. 非目标

当前版本暂不追求：

- 完全无人审校即可直接发布
- 任意低清、遮挡严重、鼠标不可见视频都高质量处理
- 复杂实拍视频与录屏视频使用同一套规则
- 第一阶段就做到高精度 UI 组件级识别

## 5. 目标用户

- 录制课程的老师
- 课程运营和教研同学
- 需要把演示视频转成 SOP 的团队
- 需要从回放中快速沉淀知识文档的内容团队

## 6. 核心使用场景

### 场景 1：课程回放转教学文档

输入课程回放链接或视频文件，输出可编辑的教学初稿、步骤清单和截图建议。

### 场景 2：软件演示转 SOP

输入产品演示录屏，输出面向新手的步骤说明和常见错误提示。

### 场景 3：老师讲解较乱的课程整理

输入操作顺序混乱、存在插话和绕路的课程视频，输出重构后的主流程文档。

## 7. 用户价值

- 大幅减少人工听课整理时间
- 保留老师真实操作依据，减少“凭空脑补”
- 提前产出可编辑初稿，缩短从视频到文档的路径
- 为后续高质量版本提供结构化中间层

## 8. 产品原则

### 8.1 语言优先

转写文本是第一事实层。截图、点击点和 UI 识别必须服务于文本理解，而不是反过来主导步骤判断。

### 8.2 事实与重构分离

必须先提取“老师实际说了什么、做了什么”，再生成“学生应该怎么读”的重构文档。

### 8.3 半自动优先

优先解决 80% 的机械工作，把结果交给人类快速修订，而不是一开始就追求全自动完美。

### 8.4 不伪精确

如果点击点、按钮名称、步骤边界不确定，必须显式标注不确定性，而不是强行给出错误的确定答案。

### 8.5 截图服务于步骤

截图不是装饰，也不是平均抽帧的副产物。每一张截图都必须服务于某一个明确步骤，并承担以下角色之一：

- 动作前定位
- 动作后确认
- 风险提示或报错说明

如果一张图无法帮助新手理解“在哪里做”和“做完后应该看到什么”，则不应进入正式教学文档。

## 9. 核心流程

### 9.1 第一阶段默认流程

1. 读取视频源
2. 生成时间戳转写
3. 基于转写与可见画面自动分步骤
4. 为每一步生成教学文字初稿
5. 为每一步生成候选截图帧
6. 对候选截图做打分、去重和角色判断
7. 为可识别操作生成点击目标候选
8. 输出审校列表，交给人工修订

### 9.1.1 第一步骤截图与点击标注子流程

对每一个已切分步骤，默认按照以下顺序执行：

1. 先根据步骤时间窗口生成候选帧，而不是全视频等间隔抽帧
2. 优先寻找动作前截图，用于定位入口
3. 再寻找动作后截图，用于确认结果
4. 对候选帧做去重，避免相邻画面几乎相同
5. 对保留候选帧做 OCR、视觉差分和界面区域过滤
6. 融合鼠标轨迹、OCR 文本、语音动作词、前后变化区域，生成点击目标候选
7. 如果证据不足，则输出“疑似点击位置”并强制进入人工审核

### 9.1.2 步骤切分信号

一个步骤成立，至少满足以下条件之一：

- 用户意图发生变化
- 界面状态发生明显变化
- 老师执行了有教学意义的动作

用于切步骤的信号包括：

- 语音动作词，例如点击、选择、输入、拖动、打开、保存
- 鼠标停顿后发生点击
- 页面局部明显变化
- 弹窗出现或消失
- 表单提交后结果变化
- URL、标题、面包屑、按钮状态变化
- 成功提示或报错提示出现

### 9.2 第二阶段增强流程

1. 从步骤稿中识别主流程
2. 把解释性内容从主流程中剥离
3. 把重复讲解重组到 FAQ 或注意事项
4. 为每一步补充预期结果与风险提示

### 9.3 第三阶段增强流程

1. 基于受众选择输出版本
2. 基于模板选择文风和排版结构
3. 增强 UI 元素命名和对象级点击标注

## 10. 输入与输出

### 10.1 输入

- 视频文件
- 回放链接
- 视频流地址
- 可选：字幕、转写稿、抽帧、课程简介

### 10.2 第一阶段输出

- 转写稿或转写整理稿
- 步骤清单
- 截图建议
- 点击点建议
- 教学初稿
- 审校问题列表

其中“截图建议”和“点击点建议”至少需要包含：

- 步骤对应的截图角色：`locator` / `result` / `warning`
- 动作前截图与动作后截图的时间点
- 候选截图被选中的原因
- 点击目标名称或回退名称
- 证据来源
- 置信度
- 是否必须人工确认

### 10.3 第二阶段输出

- 主流程文档
- 说明区块
- FAQ 区块
- 一步一动作版本

### 10.4 第三阶段输出

- 面向老师版
- 面向学生版
- 面向运营版
- 多风格导出版本

## 11. 关键数据结构

每个步骤至少需要包含：

- step_id
- start_time
- end_time
- transcript_excerpt
- title
- instruction
- expected_result
- caution
- needs_screenshot
- confidence
- draft_paragraph

每个步骤在截图与点击标注层建议拆成以下结构：

- `screenshot_candidates[]`
- `selected_screenshots[]`
- `click_targets[]`
- `evidence_signals[]`
- `review_status`
- `review_notes`

其中：

- `screenshot_candidates[]`：保存候选截图，字段至少包含 `timestamp`、`frame_id`、`role`、`reason`、`confidence`、`ui_region`
- `selected_screenshots[]`：保存最终保留截图，字段至少包含 `timestamp`、`role`、`is_primary`、`paired_step_id`
- `click_targets[]`：保存点击候选，字段至少包含 `label`、`point_or_bbox`、`confidence`、`source_signals`、`needs_review`
- `evidence_signals[]`：保存多信号证据，例如 `cursor`、`ocr`、`spoken_action`、`visual_diff`、`ui_metadata`
- `review_status`：取值建议为 `auto_pass` / `needs_review` / `human_fixed`
- `review_notes`：记录为何需要人工复核

第二阶段建议补充：

- section_type：`main_flow` / `explanation` / `faq`
- dependency_hint
- verification_hint

## 12. 核心能力拆解

### 12.1 自动转写

目标是先拿到带时间戳的语言层。转写准确率不必一开始极致，但要足够支撑后续步骤分割。

### 12.2 自动分步骤

步骤判断应优先基于语言动作词和语义子目标，再结合画面变化校验。

### 12.3 自动选截图

截图只服务于教学理解，优先覆盖：

- 定位入口
- 结果确认
- 风险提示

#### 12.3.1 可执行要求

- 不按固定时间间隔截帧，必须围绕步骤时间窗口取图
- 每个步骤默认优先保留 1 张动作前截图和 1 张动作后截图
- 如果动作前后画面几乎相同，只保留更有教学价值的一张
- 如果该步骤是纯解释、无界面变化、无定位价值，则允许无图
- 如果截图无法支撑“定位、确认、避坑”中的任一目标，则不入正文
- 如果相邻步骤截图高度重复，应合并或降级为辅助图

#### 12.3.2 候选帧打分规则

候选截图分数建议由以下因素组成：

- 是否接近动作发生点
- 是否包含清晰可识别的控件文案
- 是否包含明显的状态变化
- 是否处于教学区域，而非浏览器地址栏、桌面 Dock、摄像头角标等非教学区域
- 是否与已选截图重复

#### 12.3.3 拒选条件

以下情况默认拒绝入选正式截图：

- 只有压缩噪声变化，没有真实界面变化
- 变化主要发生在浏览器壳层、系统 Dock、摄像头小窗等区域
- 文本模糊到无法 OCR 或人工难以理解
- 截图包含敏感信息且无法安全处理

### 12.4 自动标点击点

点击点必须来自多信号融合，例如：

- 鼠标轨迹
- OCR 文本
- 语音动作词
- 前后画面变化

#### 12.4.1 输出形态

点击标注支持三种输出形态：

- 精确框：当可定位到稳定控件区域时输出 `bbox`
- 文字锚点：当知道老师点的是哪个控件，但框不够稳时输出文本框或文本锚点
- 疑似点击位置：当只有部分证据时输出低置信候选，并要求人工确认

#### 12.4.2 证据融合规则

点击目标置信度建议按以下证据叠加：

- 鼠标轨迹命中控件区域
- OCR 识别出按钮、菜单、标签或输入框文字
- 语音里出现对应动作词和对象名
- 点击前后帧差分在该区域附近集中
- 页面元数据或上下文步骤说明与候选区域一致

高置信点击目标需要至少两类强证据共同成立。

#### 12.4.3 回退与降级规则

- 如果拿不到鼠标轨迹，可退化为 `OCR + visual_diff`
- 如果 OCR 与差分都不稳定，只能输出“疑似点击位置”
- 如果候选区域冲突，必须进入人工审核
- 如果界面上存在多个同名控件，必须结合步骤上下文或语音对象词消歧
- 如果仍无法消歧，宁可不标，也不要伪造确定点击点

#### 12.4.4 点击对象命名规则

优先级从高到低为：

1. OCR 或界面可见名称
2. 语音中明确提到的控件名称
3. 语义回退名称，例如“名称输入框”“下一步按钮”“左侧版本管理入口”
4. 无法稳定命名时标为“待人工确认控件”

### 12.5 自动生成初稿

初稿应以“人类能快速改”为目标，而不是追求最终发布质量。

## 13. 成功指标

### 13.1 第一阶段指标

- 80% 以上课程能产出可编辑文字初稿
- 70% 以上步骤边界对人工而言基本可用
- 60% 以上截图建议被人工保留
- 70% 以上步骤能产出至少 1 张有教学价值的候选截图
- 关键步骤的截图配图覆盖率达到 85% 以上
- 点击标注低置信触发人工率可控，并且不以误标代替空值
- 人工整理耗时较纯手工明显下降

建议补充的量化指标：

- 截图人工替换率
- 错误截图拒绝率
- 点击命中率
- 点击误标率
- 因截图或点击误导导致的步骤返工率
- 人工修订耗时/步

### 13.2 第二阶段指标

- 主流程与说明内容分离后，可读性明显提升
- FAQ 能覆盖高频疑问与老师绕路解释
- 一步一动作版本适合直接教学使用

### 13.3 第三阶段指标

- 不同受众版本可直接复用
- UI 识别稳定性和点击命名显著提升
- 高置信点击标注覆盖率持续提升
- UI 元素识别在跨平台场景下保持稳定

## 14. 风险与难点

- 转写质量不足会连锁影响步骤划分与初稿质量
- 录屏分辨率低时，OCR 和点击点识别会显著下降
- 老师跳讲、回退、边操作边纠正时，重构逻辑容易出错
- 不同平台 UI 变化快，元素命名和识别需要持续优化
- 鼠标不可见、录制无光标时，点击点只能依赖弱证据
- 悬浮菜单、下拉框、toast 瞬间消失时，动作后截图容易缺失
- 语音与操作不同步时，动作词和画面变化可能错位
- 同屏多个可点击区域共存时，容易选错目标
- OCR 错字会导致按钮名、字段名映射错误
- 浏览器地址栏、系统 Dock、摄像头小窗等非教学区域会污染视觉差分
- 截图可能带出 token、账号、头像等敏感信息

对应降级策略：

- 无光标时降级为 `OCR + diff + human review`
- 短暂悬浮元素优先提高抽帧密度，并允许补截 warning 图
- 语音与画面错位时，以步骤窗口而非单帧时点对齐
- 多候选冲突时必须人工确认
- 非教学区域应在候选打分前先过滤
- 涉敏截图需打码、裁剪或直接弃用

## 15. 人工审核规则

以下情况必须人审：

- 点击目标置信度低于阈值
- OCR、鼠标、差分三者出现冲突
- 该步骤属于关键路径或危险操作
- 截图无法证明结果是否成功
- 点击目标与步骤文案不一致
- 候选截图主要变化发生在非教学区域

审核动作限定为：

- 保留截图
- 替换截图
- 删除截图
- 调整点击点
- 改为无标注

放行标准是：

- 截图能帮助新手定位、确认或避坑
- 点击标注不会误导用户执行
- 截图与步骤文案之间语义一致
- 不确定项已经被显式标注或人工修正

## 18. 本轮结论

本产品当前最重要的策略调整是：

不要再把抽帧当作起点，而要把“自动转写 -> 自动分步骤 -> 自动生成文字初稿”作为第一优先级。

截图、点击点、UI 识别，都应作为增强层接在文字链路之后。

对于操作步骤截图，必须采用“步骤定义先行、动作前后成对截图、证据融合点击标注、低置信强制人审”的策略，才能进入可交付阶段。
